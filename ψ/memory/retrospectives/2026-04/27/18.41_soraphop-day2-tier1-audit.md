## Honest Feedback

**1. Context overflow mid-audit is a real quality risk.**
The session hit context limits during the audit. The recovery was clean — a detailed summary let me pick up accurately — but there's genuine risk of finding loss when an audit spans multiple conversation windows. The summary correctly preserved what I'd found so far, but if the overflow had happened in the middle of analyzing a complex piece of code (not just between reads), the reconstruction could have missed a nuance. For security work specifically, where a half-analyzed file could look clean in summary but have a real issue, this is friction worth naming. Mitigation: shorter audit scope per session, or explicit mid-audit checkpoints written to ψ/inbox before context fills.

**2. PostCSS CVE assessment required more reading than it should have.**
CVE-2026-41305 was rated Low because PostCSS is build-time only in this app. But getting to that conclusion required: locating the postcss.config file, confirming it only uses tailwind + autoprefixer, confirming no CSS-in-JS, grep for `styled-jsx` / `createGlobalStyle`. That's four separate checks for a single CVE. The pnpm audit output doesn't tell you whether a vulnerability is build-time or runtime — it just flags the version. Every transitive CVE requires this kind of manual contextualization, which is slower and less systematic than I'd like. A structured "is this dependency used at runtime or build-time?" checklist in the audit template would reduce cognitive load here.

**3. CSP policy writing is always a guess the first time.**
The CSP I added to the Caddyfile is a starter policy with `'unsafe-inline'` for style-src and `'unsafe-eval'` for script-src (Next.js needs the latter for RSC hydration, at least until nonces are implemented). But I don't have visibility into all the third-party scripts the marketing site loads, or whether the portal makes cross-origin API calls. The policy I wrote will likely cause violations on day 1. I added a comment saying to run report-only first, which is the right call — but it means the High finding (CADDY_CSP_MISSING) isn't actually "fixed" by the PR, just started. That's honest, but it means the deploy blocker doesn't get fully resolved by merging this PR alone. ANVIL will need iteration time on the policy before it enforces.

## Lessons Learned

1. **Comment ≠ constraint**: When code comments document a security policy ("always-2FA per LD-5"), treat that as a spec to verify, not a fact to accept. Check whether the corresponding flag/guard is actually wired.

2. **CVE contextualization requires build/runtime classification**: pnpm audit doesn't tell you whether a dep is used at runtime or build-time. Every CVE triage needs an explicit check: "is this in devDependencies or dependencies?" + "does the production bundle include it?"

3. **trustHost + reverse proxy = always check header forwarding**: Any auth framework with `trustHost: true` requires verifying which forwarded headers the proxy sets — and which it allows to pass through from clients. Caddy's default is not to strip unknown `header_up` values.
